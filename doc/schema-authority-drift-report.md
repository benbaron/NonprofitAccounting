# Schema authority drift report

## Purpose

This report documents where the current database schema story differs between the three apparent schema authorities in the repository:

1. `nonprofitbookkeeping.core.Database#ensureSchema()`
2. JPA/Hibernate entity mappings under `org.nonprofitbookkeeping.model`
3. Flyway, which is present as a dependency but is not yet the operational migration authority

The goal is not to declare the final end-state immediately. The immediate goal is to make the current drift explicit so future schema changes can be made deliberately.

## Current authority status

### `Database.ensureSchema()`

`Database.ensureSchema()` is the practical deployed schema initializer and upgrader. It creates migration tables, legacy account/journal tables, canonical JPA tables, constraints, compatibility views, people/counterparty tables, operational linkage tables, banking/reconciliation structures, reporting schedule configuration, and finance posting preflight checks.

The H2 migration guide also identifies this path as the in-place migration mechanism: the migration script runs the same `Database.ensureSchema()` upgrades used by the app at startup.

### JPA / Hibernate

JPA describes the intended canonical object model under `org.nonprofitbookkeeping.model`. It includes entities such as `ChartOfAccounts`, `Account`, `Fund`, `Counterparty`, `Txn`, `TxnSplit`, `Activity`, `Merchant`, `ScheduleKind`, `ReportSection`, `AccountAlias`, `FundAlias`, `AccountReportSection`, `AccountScheduleRequirement`, `AccountSubtypeScheduleDefault`, and `FundTransfer`.

However, both `src/main/resources/META-INF/persistence.xml` and `org.nonprofitbookkeeping.persistence.Jpa` configure Hibernate with:

```text
hibernate.hbm2ddl.auto=update
```

That means JPA is not merely validating the schema. It may mutate the database opportunistically.

### Flyway

Flyway is present in the Maven dependency set, but this repository does not currently appear to contain a complete authoritative Flyway migration history such as:

```text
src/main/resources/db/migration/V1__*.sql
src/main/resources/db/migration/V2__*.sql
```

Nor is there currently an obvious `Flyway.configure(...).migrate()` startup path acting as the schema authority.

Therefore, Flyway is currently a dependency/tooling direction, not the effective schema source of truth.

## Concrete drift findings

### 1. `account` identity model differs

`ensureSchema()` creates `account` first as a legacy table keyed by:

```text
account_number VARCHAR(64) PRIMARY KEY
```

It later adds canonical columns including:

```text
id BIGINT GENERATED BY DEFAULT AS IDENTITY
code VARCHAR(64)
chart_id BIGINT
subtype VARCHAR(40)
normal_balance VARCHAR(10)
parent_id BIGINT
is_posting BOOLEAN
is_active BOOLEAN
effective_from DATE
effective_to DATE
description CLOB
```

JPA `Account`, by contrast, treats `id` as the primary key and treats `chart_id + code` as the business uniqueness rule.

Drift:

```text
ensureSchema / legacy:
  account_number is the primary key
  id is added as a unique identity column

JPA:
  id is the primary key
  chart_id + code is unique
```

This table is currently dual-keyed and mixed-mode.

### 2. `account` uniqueness differs

JPA declares a unique constraint:

```text
uq_account_code(chart_id, code)
```

The visible `ensureSchema()` DDL creates indexes for:

```text
account(id)
account(chart_id)
account(parent_id)
```

but does not clearly create the same explicit `(chart_id, code)` uniqueness rule.

Risk: duplicate canonical account codes may be allowed if Hibernate does not create or maintain that constraint in all deployed databases.

### 3. `account.chart_id` nullability differs

JPA declares `Account.chart` as mandatory:

```text
@ManyToOne(optional = false)
@JoinColumn(name = "chart_id", nullable = false)
```

`ensureSchema()` adds `chart_id BIGINT` without `NOT NULL`, and its FK uses `ON DELETE SET NULL`.

Drift:

```text
JPA:
  account.chart_id must not be null

ensureSchema:
  account.chart_id can be null
  deleting a chart can set chart_id to null
```

This is a direct mismatch between the canonical model and deployed DDL.

### 4. `account` canonical required fields are not fully enforced by `ensureSchema()`

JPA marks these fields as non-null or default-required:

```text
code
name
account_type
opening_balance
normal_balance
is_posting
is_active
```

`ensureSchema()` adds several of these columns through compatibility `ALTER TABLE` statements and backfills some values, but the deployed DDL is more permissive than the entity model.

Risk: legacy-compatible rows can violate assumptions made by canonical services.

### 5. `fund` parent/index relationship differs

JPA `Fund` declares:

```text
unique code
parent relationship through parent_id
index on parent_id
index on is_active
```

`ensureSchema()` creates `fund` with `parent_id` and a unique code index, but the parent FK/index and active index are not as clearly enforced in the explicit DDL path.

Risk: fund hierarchy queries may rely on application behavior or Hibernate side effects instead of explicit DDL.

### 6. `txn_split.activity_id` and `merchant_id` relationships differ

JPA `TxnSplit` maps optional relationships:

```text
activity_id -> activity(id)
merchant_id -> merchant(id)
```

`ensureSchema()` creates the columns, but its explicit `ensureJpaConstraints()` section adds FKs for:

```text
txn_id -> txn(id)
account_id -> account(id)
fund_id -> fund(id)
```

and does not clearly add physical FKs for `activity_id` or `merchant_id`.

Risk: orphan `activity_id` / `merchant_id` values are possible unless Hibernate or application code prevents them.

### 7. `txn_split.amount_signed` scale differs

JPA maps:

```text
amount_signed DECIMAL(19,4)
```

`ensureSchema()` also creates the column as `DECIMAL(19,4)`, but then adds:

```text
CHECK (amount_signed = ROUND(amount_signed, 2))
```

Drift:

```text
JPA column definition:
  four decimal places are representable

Database check:
  values must round exactly to two decimal places
```

This may be intentional for currency, but then the JPA precision/scale should be aligned or documented as intentionally wider storage with a cents-only constraint.

### 8. `counterparty.notes` differs

JPA `Counterparty` contains:

```text
@Lob String notes
```

The visible `ensureSchema()` creation for `counterparty` includes:

```text
display_name
kind
email
phone
is_active
```

but not `notes`.

Risk: Hibernate `update` may add the column in some environments, while pure `ensureSchema()` migration may omit it.

### 9. `merchant.notes` likely differs similarly

The persistence assessment states that `Merchant` has a JPA `notes` LOB but the schema initializer does not create a `notes` column for `merchant`.

Risk: same as `counterparty.notes`: deployed schema may depend on Hibernate update rather than explicit migration code.

### 10. Link and alias table constraints differ

The persistence assessment identifies several relationships as Java-defined but not consistently DB-enforced in the schema initializer, including:

```text
fund.parent_id
account_alias.account_id
fund_alias.fund_id
account_report_section.account_id/report_section_id
account_schedule_requirement.account_id/schedule_kind_id
account_subtype_schedule_default.schedule_kind_id
fund_transfer references
```

Some of these have since been strengthened in `ensureJpaConstraints()`, especially fund-transfer-related constraints. Others should be rechecked against fresh database metadata.

### 11. `fund_transfer` is stricter in `ensureSchema()` than JPA metadata alone

JPA `FundTransfer` validates many invariants in entity callbacks.

`ensureSchema()` also adds database-level checks and constraints:

```text
status domain
posted_txn link required only when POSTED
unique posted_txn_id
positive amount
distinct source/destination funds
FKs to fund and txn
```

This is not a problem; it is an example where `ensureSchema()` is stronger than plain JPA metadata. It should be kept, but eventually expressed through versioned migration scripts.

## Operational risk summary

Current risks are:

1. Split authority: both `ensureSchema()` and Hibernate `update` can modify schema.
2. Missing Flyway authority despite Flyway dependency.
3. Legacy/canonical duplication: `journal_transaction`/`journal_entry` coexist with `txn`/`txn_split`.
4. Mixed identity domains: `account.account_number` legacy PK plus `account.id` canonical identity.
5. Constraint drift: JPA required relationships are not always enforced by DDL.
6. Environment drift: databases initialized through different paths may acquire different columns or constraints.

## Recommended policy

### Short term

Keep `Database.ensureSchema()` as the operational authority, because it is what the application and migration scripts already use.

Change Hibernate from:

```text
hibernate.hbm2ddl.auto=update
```

to:

```text
hibernate.hbm2ddl.auto=validate
```

only after the documented drift items have been resolved.

### Medium term

Introduce explicit versioned migrations for all new canonical-schema changes.

Prefer:

```text
src/main/resources/db/migration/V###__description.sql
```

over new ad hoc DDL inside `ensureSchema()`.

Use `ensureSchema()` as a compatibility wrapper/backfill coordinator during the transition.

### Long term

Make Flyway the schema authority.

Then use:

```text
Flyway migrations = create/alter schema
JPA = validate schema and provide object mapping
ensureSchema() = legacy database detection, compatibility checks, and targeted backfills only
```

## Suggested next implementation sequence

1. Add a schema drift test/tool that initializes a fresh database through `ensureSchema()`, inspects metadata, and compares it against expected canonical tables/columns/constraints.
2. Fix `counterparty.notes` and `merchant.notes` in `ensureSchema()` or remove those fields from entities if intentionally unused.
3. Align `txn_split.amount_signed` scale with the cents-only check or document the reason for `DECIMAL(19,4)` plus two-decimal enforcement.
4. Add explicit DDL for missing canonical uniqueness rules such as `uq_account_code(chart_id, code)` after duplicate-data preflight.
5. Add missing FKs and indexes for `txn_split.activity_id`, `txn_split.merchant_id`, fund hierarchy, aliases, and link tables where data is clean.
6. Move Hibernate to `validate` once the deployed schema matches the JPA model.
7. Start writing new schema changes as versioned Flyway migrations.

## Recommended immediate PR after this report

Add an automated `SchemaAuthorityDriftTest` or `SchemaMetadataSnapshotTool` that produces a machine-readable report from a fresh H2 database. The report should include:

```text
tables
columns
primary keys
foreign keys
unique constraints
indexes
check constraints where available
```

Then future changes can verify that `ensureSchema()`, JPA, and eventual Flyway migrations are moving toward one authoritative model rather than further apart.
